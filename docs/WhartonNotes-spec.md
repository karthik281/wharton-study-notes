# WhartonNotes — Phased Implementation Spec

> A private web app that publishes the WEMBA Term 4 Obsidian vault (Markdown +
> `[[wikilinks]]` + graph + tags) to a URL, with keyword search, gated access,
> and an optional RAG chatbot for semantic search.
>
> **Status:** spec only — nothing implemented. Author-ready handoff.
> **Spec date:** 2026-06-15 (verify versions/pricing before building).
> **Vault source:** `C:\Users\raoka\Documents\WEMBA\Term 4`
> **Pipeline source:** `C:\Users\raoka\Documents\Ideas\Agents\Wharton Study Notes`

---

## Context

The vault already exists and is Obsidian-native: per-session notes, ~200
concept notes per course, a course MOC, Master Notes, dashboards, and a root
`Home.md`, all plain Markdown with YAML frontmatter and `[[wikilinks]]`. It's
generated by `scripts/build_obsidian_notes.py` (Anthropic API → Markdown).

Today the only way to read it is the local Obsidian app. The goal is to view it
(graph, links, search) from any device via a browser, **privately** (it derives
from copyrighted Wharton case material and lecture content — must not be public
or search-indexed), and eventually to **ask questions** against it semantically.

Build vs. buy decision (already made): use **Quartz v4** (open-source Obsidian
→ static-site generator) rather than rebuild Obsidian's renderer/graph by hand.
Quartz gives wikilinks, backlinks, graph view, and full-text search out of the
box.

## Architecture (end state)

```
build_obsidian_notes.py ──writes .md──> vault (local, Obsidian)
        │
        ▼  sync_to_quartz.py  (copies ONLY *.md, excludes attachments)
   whartonnotes/ (git repo)
     ├── content/         ← the markdown
     ├── quartz.config.ts
     └── api/chat.ts      ← Phase 3 serverless function
        │
        ▼  git push
     Vercel build (npx quartz build → static HTML + /api functions)
        │
        ▼
   Cloudflare Access (Phase 2 gate) ──> whartonnotes.<domain>
        │
   Browser: notes + graph + keyword search + chat widget (Phase 3)
        │
   chat widget → /api/chat → Voyage embed → Supabase pgvector → Claude Haiku
```

## Dependency graph

```
Phase 1 (Quartz site) ──┬──> Phase 2 (private access)
                        └──> Phase 3 (RAG chatbot)  [also needs Phase 2 to gate the API]
```

**Sequencing rationale:** Phase 1 is the foundation (nothing to gate or query
without it). Phase 2 must land **before any real notes are uploaded to a public
URL** (IP/honor-code risk, see Risks). Phase 3 depends on Phase 1 (a site to
host the widget) and on Phase 2 (the `/api/chat` endpoint must not be an open,
billable backdoor).

# Phase 1 — Quartz static site on Vercel (free)

**Goal:** the whole vault viewable at a URL with wikilinks, backlinks, graph,
and keyword search; a `git push` of changed Markdown auto-redeploys.

### Implementation details

1. **Scaffold Quartz** into a new repo `whartonnotes`:
   - `git clone https://github.com/jackyzhao/quartz && cd quartz && npm i`
   - Node.js ≥ 20 required.
2. **Content sync** — new script `scripts/sync_to_quartz.py` in the pipeline repo:
   - Walk the vault, copy **only `*.md`** into `whartonnotes/content/`,
     preserving folder structure (course / session / Concepts).
   - **Exclude:** `*.mp4`, `*.pdf`, `*.xlsx`, `*.pptx`, `*.docx`, images,
     `*.bak`, `*(consolidated archive)*`, `transcript.txt`, `materials/`.
   - Map vault root `Home.md` → Quartz `content/index.md` (Quartz uses
     `index.md` as the landing page).
   - Idempotent: clear `content/` then re-copy (or rsync-style diff).
3. **`quartz.config.ts`** — set:
   - `pageTitle: "WhartonNotes"`, `enableSPA: true`, `enablePopovers: true`
   - `baseUrl: "whartonnotes.<domain>"` (Phase 2) or the `*.vercel.app` host
   - Keep default plugins: `ObsidianFlavoredMarkdown` (wikilinks), `Latex`,
     `SyntaxHighlighting`. Default components include `Graph`, `Backlinks`,
     `Search` (FlexSearch full-text, client-side).
   - Link resolution: `markdownLinkResolution: "shortest"` to match the vault's
     Obsidian setting.
4. **Vercel project** (Hobby/free): import the GitHub repo. Framework preset
   **Other**; Build command `npx quartz build`; Output directory `public`.
   Enable auto-deploy on push to `main`.
5. **Pipeline hook:** after `build_obsidian_notes.py` finishes a course, run
   `sync_to_quartz.py` then `git -C whartonnotes add -A && commit && push`.
   (Manual to start; a file-watcher or post-build call is a follow-up.)

### Acceptance criteria

1. Visiting the deploy URL renders `index.md` (the Home page) styled by Quartz.
2. Clicking any `[[wikilink]]` (e.g. `[[FNCE 7310 - MOC]]`) navigates to that note.
3. The **Backlinks** panel on a concept note lists the sessions that link to it.
4. The **Graph** view renders nodes/edges for sessions ↔ concepts.
5. **Search** returns the FNCE note when querying a known term ("purchasing power parity").
6. The deployed repo contains **zero** `.mp4/.pdf/.xlsx/.pptx/.docx/.bak` files
   (verify: `git ls-files | grep -iE '\.(mp4|pdf|xlsx|pptx|docx|bak)$'` → empty).
7. Editing one `.md`, syncing, and pushing produces a new Vercel deploy and the
   change is live within ~3 minutes.

| Unit | `sync_to_quartz.py` include/exclude filter (md kept, mp4/pdf dropped) | +4 |
| Build | `npx quartz build` exits 0, emits `public/index.html` | +1 |
| E2E (manual) | Load URL → click wikilink → see backlinks → search a term → graph renders | +1 |

### Effort estimate

~4h: 1h Quartz scaffold + config, 1.5h `sync_to_quartz.py` (+ tests), 0.5h
Vercel import/build settings, 1h verify + fix link/path edge cases.

### Files reference

|------|--------|
| `whartonnotes/` (new repo) | Quartz scaffold |
| `whartonnotes/quartz.config.ts` | Title, baseUrl, link resolution |

# Phase 2 — Private access (Cloudflare Access preferred)

**Goal:** only you (and any classmates you allow) can load the site; it is never
publicly readable or search-indexed.

### Why this matters

The notes summarize copyrighted Wharton cases (Jaguar, REffP) and lecture
content. A public Vercel URL is crawlable by Google. Gating is non-negotiable
and must precede real content going onto any public deploy.

### Option matrix (decided: A)

| Option | Cost | Mechanism | Notes |
|--------|------|-----------|-------|
| **A. Cloudflare Access** (chosen) | Free ≤50 users | Identity gate in front of the domain | Needs the domain on Cloudflare DNS, proxied to Vercel |
| B. Vercel Password Protection | Pro (~$20/mo) | One shared password, one toggle | Simplest, but paid |
| C. Clerk + Next.js wrapper | Free ≤~10k MAU | Per-user accounts | Requires wrapping the static export in a Next.js app — heavier |

### Implementation details (Option A)

1. **Domain:** register a domain (e.g. `whartonnotes.app`, ~$10–15/yr) or use an
   existing one; add it to **Cloudflare** (free plan) so Cloudflare manages DNS.
   *(This is the one unavoidable cost in Phase 2; Cloudflare Access itself is free.)*
2. **Point the domain at Vercel** through Cloudflare:
   - In Vercel: add the custom domain to the project.
   - In Cloudflare DNS: `CNAME whartonnotes → cname.vercel-dns.com`, **proxied**
     (orange cloud). SSL/TLS mode **Full (strict)**.
3. **Cloudflare Zero Trust → Access → Applications:**
   - Add a self-hosted application for `whartonnotes.<domain>`.
   - Policy: **Allow** `email == deeptivittal@gmail.com` (add classmates as
     needed). Identity: Google or one-time PIN.
4. **Close the `*.vercel.app` backdoor:** Cloudflare Access only protects the
   custom domain. The raw `whartonnotes-xxxx.vercel.app` stays public. Mitigate
   with one of:
   - Vercel **Deployment Protection → Vercel Authentication** (free; limits to
     your Vercel account/team), and/or
   - `vercel.json` redirect of the apex `*.vercel.app` host to the custom domain,
     and/or
   - `noindex` headers as defense-in-depth (see below).
5. **Anti-indexing defense-in-depth:** add `X-Robots-Tag: noindex` response
   header (via `vercel.json` headers) and a `robots.txt` disallow. Belt and
   suspenders behind the auth wall.

### Acceptance criteria

1. Incognito request to `https://whartonnotes.<domain>` → Cloudflare Access login
   wall (no note content visible pre-auth).
2. Authenticating with an **allowed** email → site loads fully.
3. Authenticating with a **disallowed** email → access denied.
4. The underlying `*.vercel.app` URL does **not** serve note content to an
   anonymous visitor (returns Vercel auth wall or redirects to the gated domain).
5. `curl -I` of any page shows `x-robots-tag: noindex`.

### Testing plan

| Layer | What | Count |
|-------|------|-------|
| Manual | Incognito (blocked) vs allowed-email (allowed) vs disallowed-email (denied) | +3 |
| Manual | `*.vercel.app` host not anonymously readable | +1 |
| Manual | `curl -I` shows noindex header | +1 |

### Effort estimate

~2–3h: 0.5h domain + Cloudflare DNS, 0.5h Vercel custom domain, 0.5h Access app
+ policy, 0.5h close vercel.app backdoor, 0.5h verify all four access paths.

|------|--------|
| `whartonnotes/vercel.json` | `noindex` headers; optional apex→custom redirect |
| `whartonnotes/quartz.config.ts` | `baseUrl` → custom domain |

### Rollback

Remove the Access application (site reverts to public) or unproxy DNS. To fully
retreat: drop the custom domain in Vercel; site falls back to `*.vercel.app`.

### Out of scope (Phase 2)

- The chatbot endpoint's own auth (covered in Phase 3, inherits the gate).

# Phase 3 — RAG chatbot (semantic search over the notes)

**Goal:** a chat widget on the site that answers natural-language questions over
the vault and cites the source notes, without exposing API keys.

### Architecture

```
Chat widget (Quartz component)
   │  POST /api/chat { question }
   ▼
Vercel Serverless Function  api/chat.ts        (keys server-side only)
   │ 1. embed question     → Voyage AI
   │ 2. similarity search  → Supabase RPC match_chunks (pgvector)
   │ 3. build grounded prompt with top-k chunks + their note paths
   │ 4. generate           → Claude Haiku (claude-haiku-4-5)
   ▼
{ answer, citations:[{title, path}] }  → rendered in widget
```

### Data model (Supabase Postgres + pgvector)

```sql
create extension if not exists vector;

create table note_chunks (
  id          bigint generated always as identity primary key,
  note_path   text not null,          -- e.g. "FNCE 7310 .../01 ....md"
  note_title  text not null,          -- frontmatter title
  heading     text,                   -- section heading the chunk came from
  content     text not null,          -- the chunk text
  content_hash text not null,         -- sha256(content) for incremental re-embed
  embedding   vector(1024),           -- MUST match Voyage model output dim
  updated_at  timestamptz default now()
);

create index on note_chunks using hnsw (embedding vector_cosine_ops);
create unique index on note_chunks (note_path, content_hash);

-- similarity search RPC
create or replace function match_chunks(
  query_embedding vector(1024),
  match_count int default 8
) returns table (note_path text, note_title text, heading text,
                 content text, similarity float)
language sql stable as $$
  select note_path, note_title, heading, content,
         1 - (embedding <=> query_embedding) as similarity
  from note_chunks
  order by embedding <=> query_embedding
  limit match_count;
$$;
```

> `vector(1024)` is a placeholder — set it to the actual output dimension of the
> chosen Voyage model and keep query + stored dims identical.

### Ingestion / embedding pipeline (`scripts/embed_notes.py`, new)

1. For each `*.md` in the vault, strip frontmatter, split body by H2/H3 headings
   into chunks (~500–1000 tokens, with note title + heading prepended for context).
2. Compute `content_hash`; skip chunks already in `note_chunks` with that hash
   (incremental — only changed/new chunks are re-embedded).
3. Batch-embed new chunks via **Voyage AI** (e.g. `voyage-3.5`); upsert rows.
4. Delete rows whose `note_path` no longer exists or whose hash changed (prune
   stale chunks).
5. Run after `build_obsidian_notes.py` / `sync_to_quartz.py` (same trigger point).

> **Embeddings provider note:** Anthropic does not offer an embeddings model;
> Voyage AI is Anthropic's recommended provider. Generation uses Claude; only
> embeddings use Voyage.

### `api/chat.ts` (Vercel serverless function, server-side keys)

- Input: `{ question: string }`. Validate non-empty, cap length.
- Embed `question` via Voyage → `query_embedding`.
- `supabase.rpc('match_chunks', { query_embedding, match_count: 8 })`.
- Build prompt: a system prompt instructing the model to **answer only from the
  provided context**, say "not covered in the notes" if absent, and cite the
  notes it used. User message = question + concatenated chunks (with `note_path`).
- Call Claude `claude-haiku-4-5` (verify current id). Optionally stream.
- Return `{ answer, citations }`; widget renders citations as links to the
  corresponding Quartz pages.

### Chat widget

- A Quartz custom component (floating button + panel) that POSTs to `/api/chat`
  and renders the answer + citation links. Lives in the Quartz components dir and
  is added to the layout.

### Secrets (Vercel env vars — server-side, never shipped to client)

`VOYAGE_API_KEY`, `ANTHROPIC_API_KEY`, `SUPABASE_URL`,
`SUPABASE_SERVICE_ROLE_KEY`. The `/api/chat` route reads them at runtime; the
client only ever sees the function's JSON response.

### Acceptance criteria

1. Asking "What did the FNCE professor say about why hedging may not add value?"
   returns an answer grounded in the FNCE notes **with at least one citation link**
   that opens the cited note.
2. Asking something absent from the notes returns "not covered in the notes"
   (no hallucinated answer).
3. The browser network tab shows **no** API keys — only the `/api/chat` response.
4. Re-running `embed_notes.py` after editing one note re-embeds **only** the
   changed chunks (row count for unchanged notes is stable; verify via `updated_at`).
5. The `/api/chat` endpoint is **not** callable anonymously from outside the
   Cloudflare Access gate (inherits Phase 2 protection) and rejects empty/oversized input.

| Layer | What | Count |
|-------|------|-------|

| Unit | `embed_notes.py` skips unchanged hashes, prunes deleted notes | +2 |

### Effort estimate

~10–14h: 3h `embed_notes.py` (chunk + incremental + Voyage), 1h Supabase schema
+ RPC, 4h `api/chat.ts` (retrieval + prompt + Claude), 3h chat widget UI, 2h
tests + abuse/rate-limit hardening.

### Cost model

- Embeddings: one-time per chunk + on change (Voyage, cheap; cents for the corpus).
- Per chat answer: a few cents at most on Claude Haiku.
- Supabase free tier covers pgvector for a personal vault.
- Vercel Hobby includes serverless functions (free).

### Rollback

Remove the chat widget from the Quartz layout (site reverts to static + keyword
search). Drop the Supabase table. Delete the env vars and `/api/chat`. No effect
on the notes or Phases 1–2.

### Out of scope (Phase 3)

- Multi-user chat history / accounts.
- Cross-course "study buddy" features beyond Q&A with citations.
- Fine-tuning or training on the notes (RAG only).

## Risks & mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Copyrighted/IP content on a public URL during Phase 1 | High | Do not upload real notes until Phase 2 gates the site; or enable Vercel Authentication on the very first deploy; test with dummy content first |
| `*.vercel.app` backdoor bypasses Cloudflare Access | Medium | Vercel Deployment Protection + apex redirect + `noindex` (Phase 2 AC #4) |
| `/api/chat` abused → API bill | Medium | Endpoint behind Access; add rate limiting + input caps (Phase 3 AC #5) |
| Wikilinks break if filenames change | Medium | Keep `markdownLinkResolution: "shortest"`; AC #2 covers link nav |
| Embedding dim mismatch (query vs stored) | Low | Single `EMBED_DIM` constant shared by `embed_notes.py` and the schema |
| Model id drift (`claude-haiku-4-5`, Voyage model) | Low | Verify current ids at build time; keep in one config constant |

## Open decisions for the implementer (none blocking)

1. Custom domain name + registrar (Phase 2). Default: buy `whartonnotes.app` on Cloudflare.
2. Voyage model + exact embedding dimension (Phase 3). Default: `voyage-3.5`, set `vector` dim to match.
3. Chunk size / heading granularity (Phase 3). Default: split on H2/H3, ~500–1000 tokens.
4. Whether to stream chat responses or return whole (Phase 3). Default: whole first, stream as polish.

## Definition of done (whole project)

1. Phase 1 AC 1–7 pass on the live URL.
2. Phase 2 AC 1–5 pass; site is not anonymously readable and not indexable.
3. Phase 3 AC 1–5 pass; chat answers are grounded and cited, keys never client-exposed.
4. A single documented command/sequence regenerates notes → syncs → redeploys →
   re-embeds, so adding a course updates the site and the chatbot end to end.
